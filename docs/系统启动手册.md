## 1. 系统概览

| 项目 | 内容 |
|------|------|
| 系统名称 | 莱钢集团 AI 文档比对系统（一期） |

## 2. 环境要求

### 2.1 宿主机依赖

| 软件 | 最低版本 | 说明 |
|------|---------|------|
| Docker | ≥ 26.0 | 容器运行时 |
| Docker Compose | ≥ 2.24 | 服务编排（`docker compose` 命令） |
| 磁盘空间 | ≥ 30 GB | 含模型权重约 2 GB、数据库、日志 |
| 内存 | ≥ 8 GB | BGE-M3 CPU 推理需要约 4 GB |
| CPU | ≥ 4 核 | 向量化推理建议 8 核以上 |

### 2.2 外部服务依赖

| 服务 | 说明 | 必须 |
|------|------|------|
| 通义千问 API | 主 LLM，需申请 DashScope API Key | 生产必须 |
| DeepSeek API | 备用 LLM | 可选 |
| 互联网连接 | 首次启动需下载 BGE-M3 模型（约 2 GB） | 首次启动必须 |

> **开发环境可不配置真实 LLM Key**，设置 `LLM_API_MOCK=true` 即可。

## 3. 服务架构与依赖关系

### 3.1 服务清单

| 容器名 | 镜像 | 对外端口 | 说明 |
|--------|------|---------|------|
| `lgdoc-backend` | 本地构建 | `127.0.0.1:8000` | FastAPI 后端 |
| `lgdoc-frontend` | 本地构建 | 无（经 Nginx 代理） | Vue 3 前端 |
| `lgdoc-nginx` | nginx:1.25-alpine | `80`, `443` | 反向代理网关 |
| `lgdoc-mysql` | mysql:8.0 | 无（生产）/ `127.0.0.1:3306`（开发） | 关系型数据库 |
| `lgdoc-redis` | redis:7.2-alpine | 无（生产）/ `127.0.0.1:6379`（开发） | 缓存 |
| `lgdoc-minio` | minio/minio:latest | `127.0.0.1:9001`（控制台） | 对象存储 |
| `lgdoc-milvus` | milvusdb/milvus:v2.4.0 | 无 | 向量数据库 |
| `lgdoc-milvus-etcd` | coreos/etcd:v3.5.5 | 无 | Milvus 元数据存储 |

### 3.2 启动依赖顺序

```
lgdoc-milvus-etcd ─┐
lgdoc-minio ───────┼─→ lgdoc-milvus ─┐
lgdoc-mysql ───────┤                  ├─→ lgdoc-backend ─→ lgdoc-nginx
lgdoc-redis ───────┘                  │
lgdoc-frontend ────────────────────────┘
```

> `lgdoc-backend` 依赖 MySQL、Redis 的健康检查通过后才启动；依赖 MinIO、Milvus 服务启动即可（不等健康检查，内部有重试逻辑）。

### 3.3 持久化数据卷

| 卷名 | 挂载路径 | 说明 |
|------|---------|------|
| `lgdoc-mysql-data` | 容器内 `/var/lib/mysql` | MySQL 数据文件 |
| `lgdoc-redis-data` | 容器内 `/data` | Redis AOF 持久化 |
| `lgdoc-minio-data` | 容器内 `/data` | 上传文档与报告文件 |
| `lgdoc-milvus-data` | 容器内 `/var/lib/milvus` | 向量索引数据 |
| `lgdoc-etcd-data` | 容器内 `/etcd` | Milvus 元数据 |

> **警告：** 执行 `docker compose down -v` 将删除所有数据卷，数据不可恢复。

## 4. 快速启动（开发环境）

### 4.1 第一步：克隆与配置环境变量

```bash
# 进入项目目录
cd /home/laigang/laigangai

# 复制环境变量模板
cp .env.example .env
```

用编辑器打开 `.env`，按需修改以下字段：

```bash
# 开发环境最低配置（其余保持默认值即可）

# LLM：开发时使用 Mock，无需真实 Key
LLM_API_MOCK=true
LLM_API_KEY_PRIMARY=          # 开发时留空
LLM_API_KEY_BACKUP=           # 开发时留空

# JWT：生成随机密钥（≥32字符）
JWT_SECRET_KEY=your-random-secret-key-at-least-32-chars

# 其余保持 .env.example 默认值
```

生成随机 JWT 密钥的方法：

```bash
python3 -c "import secrets; print(secrets.token_hex(32))"
# 或
openssl rand -hex 32
```

### 4.2 第二步：启动所有服务

```bash
# 开发环境：使用两个 Compose 文件叠加
docker compose -f docker-compose.yml -f docker-compose.dev.yml up
```

**首次启动耗时较长**（约 5～15 分钟），主要等待：
- MySQL 初始化数据库表结构
- Backend 镜像构建（含 Python 依赖安装）
- BGE-M3 模型文件下载（约 2 GB，仅首次）

**期望日志输出：**

### 4.3 第三步：验证服务正常

## 5. 环境变量配置

所有配置通过 `.env` 文件注入，参考 `.env.example`。

### 5.1 必填变量（无默认值，启动前必须设置）

| 变量名 | 示例值 | 说明 |
|--------|--------|------|
| `DATABASE_URL` | `mysql+aiomysql://lgdoc:changeme@lgdoc-mysql:3306/lgdoc` | MySQL 连接串（异步驱动） |
| `REDIS_URL` | `redis://:changeme@lgdoc-redis:6379/0` | Redis 连接串 |
| `MINIO_ENDPOINT` | `lgdoc-minio:9000` | MinIO 地址 |
| `MINIO_ACCESS_KEY` | `minioadmin` | MinIO 访问密钥 |
| `MINIO_SECRET_KEY` | `changeme` | MinIO 密钥 |
| `JWT_SECRET_KEY` | *(随机生成 ≥32 字符)* | JWT 签名密钥，务必保密 |

### 5.2 LLM API 配置（关键安全约束）

```bash
# 主 LLM：通义千问（DashScope）
LLM_PRIMARY_PROVIDER=qianwen
LLM_API_KEY_PRIMARY=sk-xxxxxxxxxxxxxxxx   # 从阿里云控制台获取

# 备用 LLM：DeepSeek
LLM_API_KEY_BACKUP=sk-xxxxxxxxxxxxxxxx    # 从 DeepSeek 官网获取

# Mock 模式（开发/测试专用，生产禁止设为 true）
LLM_API_MOCK=false
```

> **安全警告：** API Key 只能写在 `.env` 文件中，`.env` 已加入 `.gitignore`。禁止写入代码、日志、注释。

### 5.3 可选配置（有默认值）

| 变量名 | 默认值 | 说明 |

| `MILVUS_HOST` | `lgdoc-milvus` | Milvus 地址 |

### 5.4 MySQL 密码变量

以下变量控制 MySQL 容器本身的初始化密码（在 `docker-compose.yml` 中使用）：

```bash

MYSQL_PASSWORD=changeme             # lgdoc 用户密码（需与 DATABASE_URL 保持一致）
REDIS_PASSWORD=changeme             # Redis 密码（需与 REDIS_URL 保持一致）
```

> 将上述变量补充到 `.env` 文件中（`.env.example` 中已预留注释位置）。

## 6. 生产环境部署

### 6.1 SSL 证书准备

生产环境必须配置 HTTPS，证书放入 `infra/nginx/ssl/` 目录：

```bash
# 方式一：使用真实 CA 证书（推荐）
cp /path/to/your.crt infra/nginx/ssl/server.crt
cp /path/to/your.key infra/nginx/ssl/server.key

# 方式二：自签名证书（仅测试）
openssl req -x509 -newkey rsa:4096 -nodes \
    -out infra/nginx/ssl/server.crt \
    -keyout infra/nginx/ssl/server.key \
    -days 365 \
    -subj "/C=CN/ST=Shandong/L=Laiwu/O=LaiGang/CN=your-domain.com"
```

### 6.2 修改生产环境变量

编辑 `.env` 文件，修改以下配置为生产值：

```bash
# 生产环境必须修改
APP_ENV=prod
LLM_API_MOCK=false              # 生产环境必须为 false
LOG_LEVEL=INFO

# 改用强密码
MYSQL_ROOT_PASSWORD=<强随机密码>
MYSQL_PASSWORD=<强随机密码>
REDIS_PASSWORD=<强随机密码>
MINIO_SECRET_KEY=<强随机密码>
JWT_SECRET_KEY=<随机64字符>

# 配置真实域名的 CORS
CORS_ORIGINS=https://your-domain.com

# 配置真实 LLM Key
LLM_API_KEY_PRIMARY=sk-<真实Key>
LLM_API_KEY_BACKUP=sk-<真实Key>
```

## 7. 数据库迁移

### 7.1 迁移工具

使用 **Alembic** 管理数据库 Schema 变更，迁移文件位于 `backend/alembic/versions/`。

### 7.2 常用迁移命令

在 `backend/` 目录下执行（需已激活虚拟环境或通过 `uv run`）：

```bash
cd backend

# 查看当前数据库版本
uv run alembic current

# 应用所有待执行迁移（升级到最新）
DATABASE_URL="mysql+aiomysql://lgdoc:changeme@localhost:3306/lgdoc" \
uv run alembic upgrade head

# 回滚最近一次迁移
uv run alembic downgrade -1

# 查看迁移历史
uv run alembic history

# 生成新迁移（检测模型变更）
uv run alembic revision --autogenerate -m "描述本次变更"
```

> **注意：** 生产环境部署前必须执行 `alembic upgrade head`；开发环境（`APP_ENV=dev`）启动时会自动建表，无需手动执行。

## 10. 常见问题排查

### 10.1 Backend 启动失败：数据库连接拒绝

**现象：**
```
sqlalchemy.exc.OperationalError: (pymysql.err.OperationalError)
(2003, "Can't connect to MySQL server on 'lgdoc-mysql'")
```

**原因与解决：**

```bash
# 检查 MySQL 是否 healthy
docker compose ps lgdoc-mysql
# 若状态非 "healthy"，查看 MySQL 日志
docker compose logs lgdoc-mysql

# MySQL 初始化需要约 30 秒，backend 会自动等待
# 若持续失败，手动重启 backend
docker compose restart lgdoc-backend
```

### 10.2 Backend 卡在"加载 BGE-M3 模型"

**现象：** 容器启动后十几分钟无响应

**原因：** 首次运行需从 Hugging Face 下载约 2 GB 的模型文件

**解决：**

```bash
# 查看下载进度
docker compose logs -f lgdoc-backend
# 正常时会看到下载百分比输出

# 若网络受限，可提前在宿主机下载模型
python3 -c "
from huggingface_hub import snapshot_download
snapshot_download('BAAI/bge-m3', local_dir='./bge-m3-model')
"
# 然后修改 Dockerfile 将模型路径挂载到容器
```

### 10.3 前端无法访问后端 API（CORS 错误）

**现象：** 浏览器控制台显示
```
Access to XMLHttpRequest at 'http://localhost:8000/api/...' blocked by CORS policy
```

**原因：** `.env` 中 `CORS_ORIGINS` 未包含前端地址

**解决：**

```bash
# 检查当前配置
grep CORS_ORIGINS .env

# 添加前端地址
# 修改 .env：
CORS_ORIGINS=http://localhost:5173

# 重启 backend
docker compose restart lgdoc-backend
```

### 10.4 文件上传失败：413 Request Entity Too Large

**现象：** 上传大文件时返回 HTTP 413

**原因：** Nginx `client_max_body_size` 限制（已设为 210m）或文件超过 `MAX_FILE_SIZE_MB`

**解决：**

```bash
# 检查文件大小是否超过 200MB
ls -lh your-file.docx

# 若确需调整上限，修改 .env
MAX_FILE_SIZE_MB=300

# 同步修改 infra/nginx/nginx.conf
# client_max_body_size 310m;

docker compose restart lgdoc-backend lgdoc-nginx
```

### 10.5 Milvus 连接失败

**现象：** 后端日志出现 `Milvus 连接失败`

**原因：** Milvus 依赖 etcd 和 MinIO，启动较慢（约 1 分钟）

**解决：**

```bash
# 等待 Milvus 完全就绪
docker compose logs -f lgdoc-milvus
# 看到 "Milvus Proxy successfully started" 后再重试

# 手动重启 backend（Milvus 启动后）
docker compose restart lgdoc-backend
```

### 10.6 LLM API 调用超时

**现象：** 比对任务长时间停在"分析中"，后端日志出现 timeout

**原因：** LLM API 网络不稳定或 Key 失效

**解决：**

```bash
# 开发环境：切换为 Mock 模式
# 修改 .env：
LLM_API_MOCK=true
docker compose restart lgdoc-backend

# 生产环境：检查备用 LLM 是否配置
grep "LLM_API_KEY" .env

# 测试 API 连通性
docker exec lgdoc-backend curl -I https://dashscope.aliyuncs.com
```

### 10.7 磁盘空间不足

**现象：** 容器启动失败，或上传文件报错

**解决：**

```bash
# 查看磁盘使用
df -h
docker system df

# 清理 Docker 未使用资源（不影响运行中服务）
docker system prune -f

# 清理旧的镜像
docker image prune -a -f
```
